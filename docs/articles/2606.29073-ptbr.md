# Aprendizados — *From Tool Connection to Execution Control: Benchmarking Security Invariants in MCP-Style Agent Runtimes*

> **Ting Liu (SymbolicLight Research)** · arXiv:[2606.29073](https://arxiv.org/abs/2606.29073) · preprint (jun/2026)
> **Ângulo:** estudar — entender em detalhe a tese de que MCP resolve a *connection layer* mas não a *execution layer*, e como o HCP torna os invariantes de segurança testáveis.
> **Recorte:** foco na semântica de autorização e nos 10 casos do benchmark. Deixei de fora detalhes de implementação do repositório (`run-benchmark.mjs` etc.) e a fundamentação histórica em capabilities/IFC além do necessário para o argumento.

---

## TL;DR

MCP-style protocols resolvem bem a **connection layer** (como um agente alcança tools, resources, prompts e transports via JSON-RPC), mas deixam a **execution layer** — o que acontece *depois* que a conexão existe — espalhada entre clients, servers, prompts, approval dialogs, OAuth e logs. O paper define **8 security invariants** e implementa o **HCP (Handle-Capability Protocol)**, um reference runtime in-memory que insere um broker entre a intenção do agente e a execução no provider. Num benchmark de 10 casos: o baseline naive (B0) deixa passar 10/10 ataques, o baseline com boas práticas (B1) deixa passar 6/10, e o HCP (B2) bloqueia 10/10 preservando audit evidence. A restrição que carrega o resto: é um **invariant-coverage benchmark** in-memory, não uma medição de prevalência de exploits nem um sistema de produção.

---

## O conceito central: connection vs. execution

O artigo martela uma distinção o tempo todo:

- **Connection layer** (o que o MCP faz bem): define *como* um host alcança um server — clients, servers, tools, resources, prompts, mensagens JSON-RPC, HTTP transports, negociação de versão, OAuth para tokens. É o "encanamento".
- **Execution layer** (o gap): o que acontece *depois* que a conexão existe e o agente vai efetivamente agir. Quem autorizou *esta* ação? Sobre qual resource? O dado que saiu pode ir para onde o agente quer mandar?

Sem uma execution layer, essas decisões ficam sem um único ponto onde **principal + resource + capability + approval + handle owner + data flow + audit** sejam checados juntos, de uma vez.

**Nota terminológica** (o autor faz questão): usa **"MCP-style"** para a arquitetura geral de tool-connection e **"MCP-like"** para os baselines/shims/fixtures simplificados que ele mesmo construiu. Isso delimita a claim: ele **não** afirma que todo deployment MCP real é inseguro.

## O exemplo motivador, dissecado

O exemplo da Seção 1 combina **quatro failure modes** ao mesmo tempo:

> A description de uma tool embute instruções operacionais pedindo ao modelo que use uma segunda tool sensível. O usuário aprovou uma session ampla, mas não há grant para o target resource. A primeira tool retorna texto com dados sensíveis, e o agente tenta encaminhar (pipe) essa saída para uma external-send capability.

As quatro falhas embutidas:

1. **Metadata virou instrução** — a description (que deveria ser só descritiva) manda o modelo agir → *tool poisoning*.
2. **Approval ampla ≠ authorization específica** — o "sim" para a session inteira não é um grant para *aquele* resource.
3. **Dado sensível como string solta** — a saída vira texto livre reutilizável.
4. **Fluxo não autorizado** — o dado é encaminhado para external-send (exfiltração).

Como cada modo reage:

- **B0 (naive):** metadata + approval bastam → **ataque passa**.
- **B1 (practice-informed):** warnings, metadata linting, per-call prompts reduzem risco, mas **não vinculam** a chamada a principal + canonical resource + capability id + source handle owner → ainda **passa**.
- **B2 (HCP):** nega, a menos que *tudo* seja verdadeiro ao mesmo tempo: principal tem grant que casa com o resolved resource e a capability; o high-risk effect tem approval; o source handle pertence ao caller; a target capability aceita a data class; e a decisão é registrada.

**Insight-chave:** a força do HCP aqui não é "detectar que a description é maliciosa". É que a sensitive capability simplesmente **não tem grant** — então não importa se o texto malicioso é convincente. A authorization independe do conteúdo.

## Threat model e trust boundaries (Seção 3)

**Ativos protegidos:** user resources, tool invocation authority, server-held credentials, intermediate tool outputs, handle contents, e a própria audit evidence.

**O que o attacker pode fazer:** controlar uma tool description; operar/comprometer um MCP-like server; injetar conteúdo em tool outputs; induzir o modelo a pedir broad approvals; enviar mensagens antes da initialization ou com session state desatualizado; explorar mismatch entre a resource string do caller e a resolução real do provider.

**Três níveis de confiança (Figura 2):**

| Nível | Componentes | Papel |
|-------|-------------|-------|
| **Trusted core** | runtime, policy engine, grant store, data vault, audit log | Assumidos confiáveis — a base |
| **Partially trusted** | providers | Implementam capabilities e resolvem resources, mas o runtime impõe os checks em volta deles |
| **Untrusted** | o **model** e os **natural-language metadata** | Propõem ações, nunca decidem authorization |

Frase-síntese: *"O model e os provider metadata propõem ações; o runtime decide se essas ações são autorizadas."*

**O que o benchmark NÃO mede:** attacker que compromete o trusted runtime process; malware detection em binários locais; suficiência do sandbox de referência para shell/browser/network arbitrários. **Non-goals:** não prova que todo MCP é vulnerável, não substitui OAuth/sandbox/attestation, não é production-ready.

## Os 8 security invariants (o coração do paper)

| ID | Invariant | Comportamento esperado do runtime |
|----|-----------|-----------------------------------|
| I1 | Tool metadata is not policy authority | Descriptions/prompts não concedem permissão nem sobrepõem policy |
| I2 | Approval cannot bypass grant | High-risk actions exigem grant **E** approval |
| I3 | Caller resource cannot override provider canonical resource | Policy usa o provider-resolved resource |
| I4 | Principal is bound to grant, task, handle, and audit | Principals errados não invocam nem inspecionam owned handles |
| I5 | Scoped capability invocation is grant-backed | Required scopes disparam match de principal + resource + capability |
| I6 | Data pipe authorizes source and target | Source ownership **e** target capability policy devem passar |
| I7 | Deny paths are auditable | Falhas de invocation/peek/pipe/policy deixam evidência |
| I8 | Session and protocol state are explicit | Missing init, versões erradas e sessions stale falham previsivelmente |

Distinções conceituais importantes:
- **Approval ≠ authorization** (I2): approval é um *segundo gate*, nunca substituto do grant.
- **Audit não previne** (I7): só torna decisões e falhas rastreáveis. Por isso o benchmark pontua *blocking* e *audit* como duas dimensões separadas.
- **I1 e I8** podem ser parcialmente aproximados por connection-layer checks; **I2 a I6** exigem um runtime-owned authorization model (é por isso que B1 falha).

## As 5 attack classes e os 10 casos (Seção 6)

5 classes × 2 casos = 10 casos. Cada caso ataca *um* path e um pequeno conjunto de invariants.

| Caso | Classe | Attack path modelado | B1 | B2 (HCP) |
|------|--------|---------------------|----|----------|
| tool-poisoning-001 | Tool poisoning | Hidden metadata pede exfiltração de secret | warning only; passa | `INSUFFICIENT_SCOPE` + audit |
| tool-poisoning-002 | Tool poisoning | Benign read metadata pede delegated external send | warning only; passa | `INSUFFICIENT_SCOPE` + audit |
| confused-deputy-001 | Confused deputy | Bob lê resource da Alice via server-held authority | server identity check bloqueia | `RESOURCE_DENIED` + audit |
| confused-deputy-002 | Confused deputy | Bob tem similar-prefix grant, pede private resource da Alice | server identity check bloqueia | `RESOURCE_DENIED` + audit |
| excessive-permission-001 | Approval fatigue | Broad approval para workspace write sem grant | prompt presente; passa | `MISSING_GRANT`; grant sem approval espera |
| excessive-permission-002 | Approval fatigue | Broad approval para external send sem grant | prompt presente; passa | `MISSING_GRANT`; grant + approval sucede |
| context-poisoning-datapipe-001 | Data pipe | Output com PII piped para target que não aceita PII | warning only; passa | target data-class denial + audit |
| context-poisoning-datapipe-002 | Data pipe | Wrong principal move handle de outro principal | warning only; passa | `HANDLE_ACCESS_DENIED` + audit |
| transport-session-compat-001 | Protocol state | HCP JSON-RPC method antes de initialize / versão errada | manual checks bloqueiam | `INITIALIZATION_REQUIRED` / version denial |
| transport-session-compat-002 | Protocol state | MCP-like shim method antes de initialize / versão errada | manual checks bloqueiam | `INITIALIZATION_REQUIRED` / version denial |

**Padrão diagnóstico** (o insight central): B1 acerta as classes de **identity** (confused deputy) e **session state** (transport) — 2/2 cada — porque connection-layer checks bastam ali. Mas falha em **tool poisoning, excessive permission e data pipe** (0/2 cada) porque exigem runtime objects (grant store, owned handles, data-pipe policy). É por isso que B1 fica em 6/10.

## HCP design por dentro (Seção 5)

### O broker
O HCP medeia toda a execução. O model enxerga só uma API pequena: capability discovery, task start, data peek, data pipe, grant management, policy explanation, audit search. Providers expõem capabilities via manifests, mas **quem decide se a chamada prossegue é o runtime**.

### O que o protótipo implementa (e não)
- **Implementado:** grant creation/revocation, provider-canonical resource resolution, task start/result/watch, handle peek/revoke, data pipe, policy decision records, audit search/export.
- **Fora de escopo:** persistent storage, real provider attestation, production sandboxing.
- **Escolha deliberada:** tudo **in-memory** — foca na *semântica*, não em durabilidade de DB. O preço é a falta de audit persistente, secrets management, identidade criptográfica de provider e recovery durável — mas nada disso é necessário para *testar* se os invariants fecham os attack paths.

### Policy decision flow (Figura 3) — a ordem importa
1. **Principal** — identifica o caller.
2. **Provider canonical resource** — se a capability declara required scopes, resolve o resource *pelo provider* (I3: caller string não sobrepõe).
3. **Grant match** — procura grant casando principal + scope + resolved resource + capability id (todos ao mesmo tempo).
4. **Approval gate** — *só depois* do grant, se high-risk. Três estados: grant+approval → allow; grant sem approval → **wait**; approval sem grant → **deny** (`MISSING_GRANT`).
5. **Provider invoke.**
6. **Owned handle** — output guardado atrás de handle com owner.
7. **Audit allow/deny.**
8. **Stable failure reason** — reason code, não falha silenciosa.

### Policy decision inspecionável
Cada decision registra: principal, capability, provider-canonical resource, required scopes, matched grant ids, approval state, allow/deny flag, e **reason code** (`missing principal`, `missing grant`, `insufficient scope`, `resource mismatch`, `capability mismatch`, `expired/revoked grant`, `missing approval`, `protocol initialization failure`). Isso permite distinguir *ataque bloqueado* de *falha acidental* — liga I7 diretamente à metodologia.

### Handles e vault
Output do provider não vai ao model como string — é guardado atrás de um **handle** com metadata (principal, capability, resource, task). Peek é principal-aware; handles revogados/expirados falham com reasons estáveis. O intermediate output vira **objeto controlado pelo runtime**, não string reutilizável.

### DataPipe (Figura 4) — o mecanismo mais original
Trata movimento de dado como um **novo authorization event**, com dois lados:
- **Source side:** provar ownership do source handle (wrong owner → deny).
- **Target side:** resolve target resource, avalia target capability policy, checa approval, e impõe **data-class compatibility** (ex.: negar PII para target que não aceita PII). Entre os lados pode haver mapping/redaction.

Contraste-chave: **mais estrito que um content filter.** Um filtro *avisa* que o texto parece sensível; o DataPipe *decide* ownership e target grant. É "raw string transfer" virando dois policy events (authorized read na source + authorized write no target). O autor admite: **não é IFC de nível de linguagem** (à la JFlow) — é a versão fraca mas prática.

### Audit como provenance
Toda decisão allow/deny é registrada. O modelo está mais perto de **runtime-generated provenance** (PROV-DM, PASS) do que de logging opcional: vincula actors, activities, resources, handles, outcomes. Filosofia herdada de Certificate Transparency: prevenção na *policy*, audit para *rastreabilidade*.

## Avaliação (Seção 7)

### 7.1 Security outcomes (resultado-âncora)

| Mode | Cases | Attack successes | Blocked | Audit complete |
|------|-------|------------------|---------|----------------|
| B0 | 10 | 10 | 0 | 0 |
| B1 | 10 | 6 | 4 | 0 |
| B2 | 10 | 0 | 10 | 10 |

Ressalva de interpretação: **não estima prevalência de exploits.** É um *invariant-coverage benchmark* — cada caso cruza uma runtime boundary específica. O valor não é "HCP bloqueia todos os ataques", e sim que **cada invariant é executável, testável e falha sob ablação dirigida**.

### 7.2 Ablation — qual mecanismo carrega o peso

| Componente desligado | Casos expostos | Audit losses |
|----------------------|----------------|--------------|
| **Grant matching** | **6** | 0 |
| Approval gate | 2 | 0 |
| Handle owner check | 1 | 0 |
| Data-pipe target policy | 1 | 0 |
| Protocol initialization gate | 2 | 0 |
| Protocol version check | 2 | 0 |
| **Deny-path audit** | 0 | **10** |

Leituras: **grant matching é de longe o mais crítico** (6/10). Os outros são especializados (cada um cobre sua boundary). A linha de **audit é qualitativamente diferente**: desligá-la expõe 0 casos mas cega 10 — prova executável de que prevenção e evidência são ortogonais.

### 7.3 Runtime overhead
Microbenchmark local in-memory, 50 iterações/op, mesmo processo Node.js após warm-up. Tudo **sub-millisecond**: policy explanation 0.021 ms, task start read 0.101 ms, data peek preview 0.004 ms, data pipe write 0.118 ms. Registrou 888 audit entries / ~371 KB. Argumento: os checks in-memory são pequenos *comparados* às latências de network/model/external-service já presentes.

### 7.4 Developer-effort proxy
Sem medir horas humanas: um proxy determinístico de config surface. Nos 10 casos: 195 config lines, 60 policy fields, 12 grant edges, 6 approval gates, 4 handle bindings, 7 protocol-state fields. Maiores contagens em data-pipe e tool-poisoning (bind dos dois lados). É medida *reproduzível de tamanho*, não claim sobre developer-hours.

### 7.5 MCP-compatible coverage
18 coverage items, todos com evidência checada: 7 MCP primitives (tools, resources, prompts, completion, roots, sampling, elicitation), 6 transport/session, 5 notification/task. Por transporte: 15 stdio, 17 HTTP, 1 SSE. **Ressalva:** denominador escopado ao reference shim — **não** é official MCP certification.

### 7.6 Operator-governance surface checks
Comparação determinística plain-config vs. HCP governance layer, 10 fixtures. Motivação: operators não pedem "mais protocol objects"; perguntam *por que este local server command é risky, onde o secret será guardado, qual tool o model vê, por que a approval foi pedida, o que aconteceu no incidente*. HCP zera os 10 governance gaps das fixtures, redige secret exposure, revela high-risk local servers, dá approval summaries e audit explanations. **Auxiliar** — não é user study.

### 7.7 Ecosystem measurement
Ancora o threat model em metadata público, **não** é vulnerability study. Snapshot 2026-06-22: 90 raw rows → 80 dedup → 40 retained. Flags mais comuns: **unclear audit story em 40 (todos)**, repo/code authority em 15, browser control em 7, external network em 7, cloud infra em 6, credential em 5, memory/profile em 5. Adjudicação conservadora: 29 README-confirmed authority surfaces, 22 audit-story absences, 15 counter-evidence, 11 not-supported, 18 ambiguous. Labels = "README-visible signal", **não** "confirmed vulnerability".

## Discussão, limitações, related work (Seções 8–11)

- **Discussão:** metadata linting, approval e session checks são úteis mas *incompletos* — não criam uma decisão única de runtime que vincule tudo. Não é argumento *contra* endurecer o MCP; a pergunta é **onde a authority é decidida depois da conexão**. HCP **não substitui OAuth** (OAuth protege authorization flows e tokens; HCP trata de *como o runtime usa a authority*). Deployment realista precisa das **duas layers**.
- **Limitações:** runtime **in-memory** (grants/audit/handles não duráveis); production exigiria durable storage, attestation/signature de manifests, sandboxing por risco, admin access control. Benchmark **adversarial mas não exaustivo** — prompt injection, malicious model, compromised OAuth clients, supply-chain e provider malware **ainda importam**. Compatibility é escopada (não é certification).
- **Related work:** (a) MCP security — Huang et al., MCPTox, Invariant Labs; (b) indirect prompt injection — InjecAgent, AgentDojo. **Diferença-chave:** essas defesas dependem de *detectar/filtrar* conteúdo ou de *model robustness*; HCP trata output/metadata como non-authoritative e impõe a *próxima invocation* na runtime boundary. (c) Raízes clássicas: capability systems + confused deputy (Hardy 1988; Rajani et al. 2016); information-flow control (Denning 1976; JFlow); provenance/audit (PROV-DM, PASS, Certificate Transparency).

## Meu takeaway

Síntese mental: **grant** = authorization; **approval** = segundo gate para risco; **canonical resource** = anti-spoofing de string; **handle** = controle sobre dado parado; **DataPipe** = controle sobre dado em movimento; **audit** = evidência forense.

O paper não vende segurança de produção — vende um **design principle**: agent systems precisam de uma execution-control layer, avaliável com *reproducible invariants*, implementada como um runtime object model em vez de prompts/descriptions/dialogs/logs espalhados. A honestidade sobre o escopo (in-memory, invariant-coverage, non-goals bem demarcados) é o que torna a claim defensável.

**Pergunta em aberto para mim:** os baselines B0/B1 são justos, ou o B1 poderia ser mais forte? E quanto o resultado in-memory se sustenta quando durabilidade, attestation e sandboxing entram (que o próprio autor diz "não serem menores")?
