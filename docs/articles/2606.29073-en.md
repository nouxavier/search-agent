# Learnings — *From Tool Connection to Execution Control: Benchmarking Security Invariants in MCP-Style Agent Runtimes*

> **Ting Liu (SymbolicLight Research)** · arXiv:[2606.29073](https://arxiv.org/abs/2606.29073) · preprint (Jun 2026)
> **Angle:** study — understand in detail the thesis that MCP solves the *connection layer* but not the *execution layer*, and how HCP makes security invariants testable.
> **Scope cut:** focus on authorization semantics and the 10 benchmark cases. Left out repo implementation details (`run-benchmark.mjs` etc.) and the historical grounding in capabilities/IFC beyond what the argument needs.

---

## TL;DR

MCP-style protocols handle the **connection layer** well (how an agent reaches tools, resources, prompts, and transports over JSON-RPC), but leave the **execution layer** — what happens *after* the connection exists — split across clients, servers, prompts, approval dialogs, OAuth, and logs. The paper defines **8 security invariants** and implements **HCP (Handle-Capability Protocol)**, an in-memory reference runtime that inserts a broker between agent intent and provider execution. In a 10-case benchmark: the naive baseline (B0) lets 10/10 attacks through, the practice-informed baseline (B1) lets 6/10 through, and HCP (B2) blocks 10/10 while preserving audit evidence. The constraint that carries the rest: it is an **invariant-coverage benchmark** that runs in-memory — not a measurement of exploit prevalence, and not a production system.

---

## The core idea: connection vs. execution

The paper hammers one distinction throughout:

- **Connection layer** (what MCP does well): defines *how* a host reaches a server — clients, servers, tools, resources, prompts, JSON-RPC messages, HTTP transports, version negotiation, OAuth for tokens. The "plumbing".
- **Execution layer** (the gap): what happens *after* the connection exists and the agent actually acts. Who authorized *this* action? Over which resource? Can the returned data flow where the agent wants to send it?

Without an execution layer, these decisions have no single point where **principal + resource + capability + approval + handle owner + data flow + audit** are checked together, at once.

**Terminology note** (the author is deliberate): uses **"MCP-style"** for the broad tool-connection architecture and **"MCP-like"** for the simplified baselines/shims/fixtures built for this artifact. This bounds the claim: he does **not** argue that every real MCP deployment is insecure.

## The motivating example, dissected

The Section 1 example combines **four failure modes** at once:

> A tool description embeds operational instructions asking the model to use a second sensitive tool. The user approved a broad session, but there is no grant for the target resource. The first tool returns text with sensitive data, and the agent tries to pipe that output into an external-send capability.

The four embedded failures:

1. **Metadata became instruction** — the description (which should be descriptive only) tells the model to act → *tool poisoning*.
2. **Broad approval ≠ specific authorization** — the "yes" for the whole session is not a grant for *that* resource.
3. **Sensitive data as a loose string** — the output becomes reusable free text.
4. **Unauthorized flow** — the data is piped into external-send (exfiltration).

How each mode reacts:

- **B0 (naive):** metadata + approval suffice → **attack succeeds**.
- **B1 (practice-informed):** warnings, metadata linting, per-call prompts reduce risk but **do not bind** the call to principal + canonical resource + capability id + source handle owner → still **succeeds**.
- **B2 (HCP):** denies unless *everything* holds at once: principal has a grant matching the resolved resource and capability; the high-risk effect has approval; the source handle belongs to the caller; the target capability accepts the data class; and the decision is recorded.

**Key insight:** HCP's strength here is not "detecting that the description is malicious". It is that the sensitive capability simply **has no grant** — so it does not matter how convincing the malicious text is. Authorization is content-independent.

## Threat model and trust boundaries (Section 3)

**Protected assets:** user resources, tool invocation authority, server-held credentials, intermediate tool outputs, handle contents, and audit evidence itself.

**What the attacker can do:** control a tool description; operate/compromise an MCP-like server; inject content into tool outputs; induce the model to ask for broad approvals; send messages before initialization or with stale session state; exploit a mismatch between the caller's resource string and the provider's actual resolution.

**Three trust levels (Figure 2):**

| Level | Components | Role |
|-------|-----------|------|
| **Trusted core** | runtime, policy engine, grant store, data vault, audit log | Assumed trusted — the foundation |
| **Partially trusted** | providers | Implement capabilities and resolve resources, but the runtime enforces the checks around them |
| **Untrusted** | the **model** and **natural-language metadata** | Propose actions, never decide authorization |

Summary line: *"The model and provider metadata propose actions; the runtime decides whether those actions are authorized."*

**What the benchmark does NOT measure:** an attacker who compromises the trusted runtime process; malware detection in local binaries; whether the reference sandbox suffices for arbitrary shell/browser/network. **Non-goals:** does not prove all MCP is vulnerable, does not replace OAuth/sandbox/attestation, is not production-ready.

## The 8 security invariants (the heart of the paper)

| ID | Invariant | Expected runtime behavior |
|----|-----------|---------------------------|
| I1 | Tool metadata is not policy authority | Descriptions/prompts cannot grant permission or override policy |
| I2 | Approval cannot bypass grant | High-risk actions require both grant **AND** approval |
| I3 | Caller resource cannot override provider canonical resource | Policy uses the provider-resolved resource |
| I4 | Principal is bound to grant, task, handle, and audit | Wrong principals cannot invoke or inspect owned handles |
| I5 | Scoped capability invocation is grant-backed | Required scopes trigger principal + resource + capability matching |
| I6 | Data pipe authorizes source and target | Source ownership **and** target capability policy must both pass |
| I7 | Deny paths are auditable | Failed invocation/peek/pipe/policy leave evidence |
| I8 | Session and protocol state are explicit | Missing init, wrong versions, stale sessions fail predictably |

Key conceptual distinctions:
- **Approval ≠ authorization** (I2): approval is a *second gate*, never a substitute for the grant.
- **Audit does not prevent** (I7): it only makes decisions and failures traceable. That is why the benchmark scores *blocking* and *audit* as two separate dimensions.
- **I1 and I8** can be partially approximated by connection-layer checks; **I2 through I6** require a runtime-owned authorization model (this is why B1 fails).

## The 5 attack classes and 10 cases (Section 6)

5 classes × 2 cases = 10 cases. Each case targets *one* path and a small set of invariants.

| Case | Class | Modeled attack path | B1 | B2 (HCP) |
|------|-------|---------------------|----|----------|
| tool-poisoning-001 | Tool poisoning | Hidden metadata asks to exfiltrate a secret | warning only; succeeds | `INSUFFICIENT_SCOPE` + audit |
| tool-poisoning-002 | Tool poisoning | Benign read metadata requests delegated external send | warning only; succeeds | `INSUFFICIENT_SCOPE` + audit |
| confused-deputy-001 | Confused deputy | Bob reads Alice's resource via server-held authority | server identity check blocks | `RESOURCE_DENIED` + audit |
| confused-deputy-002 | Confused deputy | Bob has similar-prefix grant, requests Alice's private resource | server identity check blocks | `RESOURCE_DENIED` + audit |
| excessive-permission-001 | Approval fatigue | Broad approval for workspace write without grant | prompt present; succeeds | `MISSING_GRANT`; grant without approval waits |
| excessive-permission-002 | Approval fatigue | Broad approval for external send without grant | prompt present; succeeds | `MISSING_GRANT`; grant + approval succeeds |
| context-poisoning-datapipe-001 | Data pipe | PII output piped into a target that rejects PII | warning only; succeeds | target data-class denial + audit |
| context-poisoning-datapipe-002 | Data pipe | Wrong principal moves another principal's handle | warning only; succeeds | `HANDLE_ACCESS_DENIED` + audit |
| transport-session-compat-001 | Protocol state | HCP JSON-RPC method before initialize / wrong version | manual checks block | `INITIALIZATION_REQUIRED` / version denial |
| transport-session-compat-002 | Protocol state | MCP-like shim method before initialize / wrong version | manual checks block | `INITIALIZATION_REQUIRED` / version denial |

**Diagnostic pattern** (the central insight): B1 gets the **identity** class (confused deputy) and **session state** class (transport) right — 2/2 each — because connection-layer checks suffice there. But it fails on **tool poisoning, excessive permission, and data pipe** (0/2 each) because those require runtime objects (grant store, owned handles, data-pipe policy). This is why B1 lands at 6/10.

## HCP design internals (Section 5)

### The broker
HCP mediates all execution. The model sees only a small API: capability discovery, task start, data peek, data pipe, grant management, policy explanation, audit search. Providers expose capabilities via manifests, but **the runtime decides whether a call can proceed**.

### What the prototype implements (and not)
- **Implemented:** grant creation/revocation, provider-canonical resource resolution, task start/result/watch, handle peek/revoke, data pipe, policy decision records, audit search/export.
- **Out of scope:** persistent storage, real provider attestation, production sandboxing.
- **Deliberate choice:** everything is **in-memory** — focuses on *semantics*, not DB durability. The price is no persistent audit, secrets management, cryptographic provider identity, or durable recovery — but none of that is needed to *test* whether the invariants close the attack paths.

### Policy decision flow (Figure 3) — order matters
1. **Principal** — identify the caller.
2. **Provider canonical resource** — if the capability declares required scopes, resolve the resource *through the provider* (I3: caller string cannot override).
3. **Grant match** — find a grant matching principal + scope + resolved resource + capability id (all at once).
4. **Approval gate** — *only after* the grant, if high-risk. Three states: grant+approval → allow; grant without approval → **wait**; approval without grant → **deny** (`MISSING_GRANT`).
5. **Provider invoke.**
6. **Owned handle** — output stored behind a handle with an owner.
7. **Audit allow/deny.**
8. **Stable failure reason** — a reason code, not a silent failure.

### Inspectable policy decision
Each decision records: principal, capability, provider-canonical resource, required scopes, matched grant ids, approval state, allow/deny flag, and a **reason code** (`missing principal`, `missing grant`, `insufficient scope`, `resource mismatch`, `capability mismatch`, `expired/revoked grant`, `missing approval`, `protocol initialization failure`). This distinguishes a *blocked attack* from an *accidental failure* — tying I7 directly to methodology.

### Handles and vault
Provider output does not reach the model as a string — it is stored behind a **handle** with metadata (principal, capability, resource, task). Peek is principal-aware; revoked/expired handles fail with stable reasons. Intermediate output becomes a **runtime-controlled object**, not a reusable string.

### DataPipe (Figure 4) — the most original mechanism
Treats data movement as a **new authorization event**, with two sides:
- **Source side:** prove ownership of the source handle (wrong owner → deny).
- **Target side:** resolve target resource, evaluate target capability policy, check approval, and enforce **data-class compatibility** (e.g., deny PII to a target that rejects PII). Mapping/redaction can happen between sides.

Key contrast: **stricter than a content filter.** A filter *warns* that text looks sensitive; DataPipe *decides* ownership and target grant. It turns "raw string transfer" into two policy events (authorized read at source + authorized write at target). The author concedes: **not language-level IFC** (à la JFlow) — the weaker but practical version.

### Audit as provenance
Every allow/deny is recorded. The model is closer to **runtime-generated provenance** (PROV-DM, PASS) than optional logging: it binds actors, activities, resources, handles, outcomes. Philosophy borrowed from Certificate Transparency: prevention in *policy*, audit for *traceability*.

## Evaluation (Section 7)

### 7.1 Security outcomes (anchor result)

| Mode | Cases | Attack successes | Blocked | Audit complete |
|------|-------|------------------|---------|----------------|
| B0 | 10 | 10 | 0 | 0 |
| B1 | 10 | 6 | 4 | 0 |
| B2 | 10 | 0 | 10 | 10 |

Interpretation caveat: **does not estimate exploit prevalence.** It is an *invariant-coverage benchmark* — each case crosses a specific runtime boundary. The value is not "HCP blocks all attacks", but that **each invariant is executable, testable, and fails under targeted ablation**.

### 7.2 Ablation — which mechanism carries the weight

| Disabled component | Cases exposed | Audit losses |
|--------------------|---------------|--------------|
| **Grant matching** | **6** | 0 |
| Approval gate | 2 | 0 |
| Handle owner check | 1 | 0 |
| Data-pipe target policy | 1 | 0 |
| Protocol initialization gate | 2 | 0 |
| Protocol version check | 2 | 0 |
| **Deny-path audit** | 0 | **10** |

Readings: **grant matching is by far the most critical** (6/10). The others are specialized (each covers its boundary). The **audit row is qualitatively different**: disabling it exposes 0 cases but blinds 10 — executable proof that prevention and evidence are orthogonal.

### 7.3 Runtime overhead
Local in-memory microbenchmark, 50 iterations/op, same Node.js process after warm-up. All **sub-millisecond**: policy explanation 0.021 ms, task start read 0.101 ms, data peek preview 0.004 ms, data pipe write 0.118 ms. Recorded 888 audit entries / ~371 KB. Argument: in-memory checks are small *relative to* the network/model/external-service latencies already present.

### 7.4 Developer-effort proxy
No human hours measured: a deterministic config-surface proxy. Across 10 cases: 195 config lines, 60 policy fields, 12 grant edges, 6 approval gates, 4 handle bindings, 7 protocol-state fields. Largest counts in data-pipe and tool-poisoning (binding both sides). A *reproducible size* measure, not a developer-hours claim.

### 7.5 MCP-compatible coverage
18 coverage items, all with checked evidence: 7 MCP primitives (tools, resources, prompts, completion, roots, sampling, elicitation), 6 transport/session, 5 notification/task. By transport: 15 stdio, 17 HTTP, 1 SSE. **Caveat:** denominator scoped to the reference shim — **not** official MCP certification.

### 7.6 Operator-governance surface checks
Deterministic plain-config vs. HCP governance layer comparison, 10 fixtures. Motivation: operators do not ask for "more protocol objects"; they ask *why is this local server command risky, where will the secret live, which tool does the model see, why is this approval requested, what happened after the incident*. HCP zeroes the 10 governance gaps, redacts secret exposure, surfaces high-risk local servers, gives approval summaries and audit explanations. **Auxiliary** — not a user study.

### 7.7 Ecosystem measurement
Grounds the threat model in public metadata, **not** a vulnerability study. Snapshot 2026-06-22: 90 raw rows → 80 dedup → 40 retained. Most common flags: **unclear audit story in 40 (all)**, repo/code authority in 15, browser control in 7, external network in 7, cloud infra in 6, credential in 5, memory/profile in 5. Conservative adjudication: 29 README-confirmed authority surfaces, 22 audit-story absences, 15 counter-evidence, 11 not-supported, 18 ambiguous. Labels = "README-visible signal", **not** "confirmed vulnerability".

## Discussion, limitations, related work (Sections 8–11)

- **Discussion:** metadata linting, approval, and session checks are useful but *incomplete* — they do not create a single runtime decision binding everything. Not an argument *against* hardening MCP; the question is **where authority is decided after the connection**. HCP **does not replace OAuth** (OAuth protects authorization flows and tokens; HCP is about *how the runtime uses authority*). A realistic deployment needs **both layers**.
- **Limitations:** **in-memory** runtime (grants/audit/handles not durable); production would need durable storage, manifest attestation/signature, risk-appropriate sandboxing, admin access control. Benchmark is **adversarial but not exhaustive** — prompt injection, malicious model, compromised OAuth clients, supply-chain, and provider malware **still matter**. Compatibility is scoped (not certification).
- **Related work:** (a) MCP security — Huang et al., MCPTox, Invariant Labs; (b) indirect prompt injection — InjecAgent, AgentDojo. **Key difference:** those defenses depend on *detecting/filtering* content or on *model robustness*; HCP treats output/metadata as non-authoritative and enforces the *next invocation* at the runtime boundary. (c) Classic roots: capability systems + confused deputy (Hardy 1988; Rajani et al. 2016); information-flow control (Denning 1976; JFlow); provenance/audit (PROV-DM, PASS, Certificate Transparency).

## My takeaway

Mental summary: **grant** = authorization; **approval** = second gate for risk; **canonical resource** = string anti-spoofing; **handle** = control over data at rest; **DataPipe** = control over data in motion; **audit** = forensic evidence.

The paper does not sell production security — it sells a **design principle**: agent systems need an execution-control layer, evaluable with *reproducible invariants*, implemented as a runtime object model rather than scattered prompts/descriptions/dialogs/logs. The honesty about scope (in-memory, invariant-coverage, well-marked non-goals) is what makes the claim defensible.

**Open question for me:** are the B0/B1 baselines fair, or could B1 be stronger? And how well does the in-memory result hold once durability, attestation, and sandboxing enter (which the author himself calls "not minor")?
