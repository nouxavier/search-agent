# Takeaways — *MCP Server Architecture Patterns for LLM-Integrated Applications*

> **Rodrigues & Vas** · arXiv:[2606.30317](https://arxiv.org/abs/2606.30317) · ICSME 2026 candidate
> **Angle:** building a good MCP server — not studying the paper.
> **Scope:** filtered for what has engineering consequences; κ, inter-rater agreement, and the pattern formalism were left out.

---

## TL;DR

Designing an MCP server is an *API design* problem with an unusual constraint: **the client is an LLM that picks the operation by reading the natural-language description at runtime, not by consulting the schema or the docs.** Almost every good design decision derives from this single constraint. The protocol is deliberately minimal — state, async, and sanitization are on you.

---

## The 5 rules that matter most for building

In order of impact.

### 1. A tool description is a *load-bearing* artifact — treat it as code

This is the paper's central topic. The LLM selects the tool by its description, not its schema. A vague description (or one that just repeats the name, like `send_message` with no body) makes the tool **work in testing and fail to be selected in production** — the most insidious failure mode, because it passes unit tests.

- Write: what it does, when to use it, what it returns — for someone who has never seen it.
- Review the description in PR, with the same rigor as code.
- Treat any divergence between description and behavior as a **bug**, not stale docs.

### 2. Keep ≤ ~10–15 tools per context

Practical ceiling of selection accuracy measured in production:

| Model | Drops below 90% accuracy |
|---|---|
| Claude Haiku 4.5 | between **10 and 15** tools (91% @ 10, 87% @ 15) |
| Claude Sonnet 4 | between **20 and 30** tools (≥90% up to 20) |

Design consequence: **don't dump every tool into one server.** As the catalog grows, partition and do **retrieval-over-tools** — expose only the relevant subset per request — instead of a *static-merge* of everything. Fewer well-named tools > many ambiguous ones.

> ⚠️ Caveat: the number comes from observational telemetry at **one** voice-AI company with a tight latency budget (500ms). Treat 10–15 as a **conservative floor for latency-sensitive paths**, not universal law. The mechanism (partition + retrieval) generalizes; the specific number does not. Independent work (RAG-MCP; LongFuncEval) confirms the degradation, with the knee of the curve around ~30 candidates in scenarios without latency constraints.

### 3. Decompose — never build a "God Tool"

Each distinct operation = one named tool with a precise schema. Never `do_anything(action: string, params: object)`: hiding the semantics inside a runtime argument destroys the selection signal (it's rule 1 in reverse). Neither too many tools (rule 2), nor one tool that should be ten.

#### Bad — the classic God Tool

```json
{
  "name": "manage_calendar",
  "description": "Manages calendar events",
  "inputSchema": {
    "type": "object",
    "properties": {
      "action": { "type": "string" },
      "params": { "type": "object" }
    }
  }
}
```

The LLM has to *guess* that `action` accepts `"create"|"update"|"delete"|"list"` and that `params` changes shape depending on the action. None of that is in the signal it uses to decide (name + description). It calls `manage_calendar` for anything calendar-related and then gets `params` wrong. Double failure: right selection, wrong arguments.

#### Good — decomposed per operation

```json
{
  "name": "create_calendar_event",
  "description": "Creates a new event in the user's calendar. Use when the user wants to schedule something new (meeting, reminder, appointment). Returns the ID and link of the created event. Do not use to modify existing events — use update_calendar_event.",
  "inputSchema": {
    "type": "object",
    "required": ["title", "start", "end"],
    "properties": {
      "title":  { "type": "string", "description": "Event title" },
      "start":  { "type": "string", "format": "date-time" },
      "end":    { "type": "string", "format": "date-time" },
      "attendees": { "type": "array", "items": { "type": "string", "format": "email" } }
    }
  }
}
```

Plus `update_calendar_event`, `delete_calendar_event`, `list_calendar_events` in the same style. Each tool has a typed schema (the model knows `start` is a date-time, not a free blob) and the description says *when to use* and *when not to use* it, pointing at the sibling tool. That's rule 1 and rule 3 working together.

#### The subtle God Tool — the one that fools you

It passes review because it *looks* decomposed:

```json
{
  "name": "search",
  "description": "Search the system",
  "inputSchema": {
    "type": "object",
    "properties": {
      "type":  { "type": "string", "enum": ["user", "order", "product", "invoice"] },
      "query": { "type": "string" }
    }
  }
}
```

It has an enum, it has a typed schema — but it's still a disguised God Tool. The `type` field **branches behavior at runtime**, so the LLM reasons about business semantics (searching a user vs. an invoice are different intents) inside an argument, not in the tool choice. The selection signal stays weak.

Good version:

```json
{ "name": "search_users",    "description": "Search users by name, email, or ID. Returns profile and account status." }
{ "name": "search_orders",   "description": "Search orders by number, customer, or date. Returns items, amount, and delivery status." }
{ "name": "search_invoices", "description": "Search invoices by number or customer. Returns amount, due date, and payment status." }
```

#### The deciding rule — behavior vs. filter

Don't decompose to the point of blowing past the tool ceiling (rule 2). The dividing line:

> **Does the argument change *what the tool does* (→ decompose) or just *filter the same result* (→ make it a parameter)?**

- **Decompose** when the operations have different intents, input schemas, or return values → `create_event` vs. `delete_event`. In `manage_calendar`, `action` changed what the tool does — wrong.
- **Don't decompose** when it's the same input varying only in a value without changing semantics. Don't build `list_events_today` / `list_events_this_week` / `list_events_this_month` — that's one tool `list_events(range)`, because intent and return are identical and `range` only filters — right.

The quick test for the subtle case: *if the "types"/"actions" have different descriptions, use cases, or return fields, they are different tools.*

### 4. Sanitize all external content before returning it

The only **security** item on the list. Backend/user content (a comment, a document body, a form input) arrives raw at the LLM, and text like *"Ignore previous instructions and…"* is processed as an **instruction, not data** (prompt injection). Insert an **explicit, named** sanitization layer — don't trust the backend, don't do it inline and forget. The protocol won't do this for you.

### 5. Never expose a long synchronous operation

MCP has no async callback mechanism. Any operation over a few seconds (video encoding, large-file processing) makes the client **time out**. The canonical pattern: return a `job_id` synchronously and expose a separate `poll_job(id)` tool.

## 3 structural decisions

### State — only when needed, with a budget for *reaping*

MCP is **stateless request-response by default**. If you need state across calls:

- Generate a `session_id` on connection, include it in every response, and the client carries it back on each call.
- Keep per-session context in memory — or Redis to scale horizontally.
- Expire sessions on inactivity.

The real costs you inherit: (a) the **LLM has to pass the session ID reliably, which is not guaranteed** — you're asking a probabilistic model to carry a state pointer; (b) memory leaks if sessions aren't reaped; (c) horizontal scaling requires a distributed store. **Add state only when a turn genuinely depends on the previous one**, not for convenience.

### Co-location matters more than the transport choice

Two transports: `stdio` (local, in-process) and `streamable-http` (remote, HTTP + optional SSE). The overhead difference between them is **sub-millisecond and irrelevant the moment you cross a host boundary** — network RTT (~30ms same-region) dominates by 2–3 orders of magnitude.

The decisions that actually move latency:

1. Is the server **co-located** with the client?
2. Does downstream fan-out (a proxy/aggregator) add **another network hop**? Each proxy layer doubles RTT exposure.

Decide this first; the transport encoding is a detail.

### Authentication — decide before you write the first tool

Auth on an MCP server is not "add a middleware later". The LLM-client constraint changes the threat model: the client is neither a human nor a deterministic app, it's a model that decides what to call at runtime from text that may have been injected. That's why auth has to sit at the base of the design, not at the end.

**Rule 1 — authenticate at the transport, never inside the tool handler.** `streamable-http` supports a **Bearer token** in the HTTP header. Verification happens *before* the request becomes a tool call. If you leave the auth check inside the handler (`if (!isAuthorized(req)) throw...`), you're trusting that every tool implemented the check correctly — and it only takes one forgetting. Worse: a handler that authenticates by looking at request arguments is one prompt injection away from being fooled, because the arguments come from the LLM. Auth at the edge, identity resolved before any business logic.

**Rule 2 — scope tokens per tool set, not per whole server.** A token should not grant access to everything the server exposes. Model scopes as capabilities: a read token accesses only the read tools/resources; a write token, an explicit subset. This limits the *blast radius* when a token leaks (and tokens leak — they live in client config, in logs, in screenshots). In multi-tenant, the scope also carries the tenant: the token identifies *which* tenant, and the server never derives the tenant from an argument passed by the LLM.

**Rule 3 — the authenticated identity is the caller's, not the end user's.** This is where the most common confusion lives. What authenticates at the transport is the *client application / agent*, not necessarily the person behind it. If your tools act on behalf of a user (reading *their* calendar, sending email *as* them), you need a second layer: the transport token says "this agent may talk to me"; a user context (delegated OAuth, a forwarded user session token) says "and it's acting on behalf of person X". Don't collapse the two. An authenticated agent acting with the wrong user's privilege is privilege escalation.

**Rule 4 — auth never depends on content that passed through the LLM.** An authorization decision rests on: the token (transport), the scope (what it allows), and, when applicable, the delegated user context. *Never* on a tool argument, a request field assembled by the model, or text coming from a resource. Everything that crosses the LLM is potentially adversarial. This is the point where MCP auth differs from traditional API auth — the "client" is manipulable by injection.

**stdio is a different trust model.** With the `stdio` transport the server runs as a local subprocess of the client, inheriting the permissions of the process/user that launched it — there's no Bearer token in the middle. The security boundary there is the OS, not the protocol. That's adequate for local tools (accessing the user's own files), but it means "going remote" (streamable-http) isn't just swapping transports: it's adding the entire auth layer that stdio let you skip. Decide early whether the server will be remote, because it changes the whole security design.

**Where the Proxy Aggregator comes in.** At scale, the aggregator is the natural place to centralize auth: a single point of token issuance/validation, scope mapping, and audit for the whole fleet, instead of each upstream server reinventing it. The cost is that it also becomes a concentrated target — compromising the aggregator compromises access to the fleet. Treat it as a first-class security component (credential rotation, isolation, HA), not a dumb router.

### The two identity layers — agent vs. user

This is the most expensive auth mistake to fix later, because collapsing the two layers appears to work in development and only blows up when there's more than one real user. Separate them from the start.

**Layer 1 — may the agent talk to me?** It's the transport's Bearer token (section above). It authenticates the *client application / agent* — the process that opened the MCP connection. It answers: "is this thing calling me a client I recognize and authorize to talk?". It's binary and per-connection.

**Layer 2 — on whose behalf is it acting?** It's the **delegated user context**: an OAuth on behalf of the user, or a forwarded user session token in the call. It answers: "and are the actions it's requesting on behalf of person X, with person X's *privileges*?". It's per-user and changes with every person the agent serves.

> The transport token says **"this agent may talk to me"**. The user context says **"and it's acting on behalf of person X"**. Don't collapse the two.

**Why collapsing is privilege escalation.** Picture an assistant that reads the calendar and sends email. If you only have Layer 1, the agent authenticates with *one* token and all tools run with *one* privilege — probably that of a broad service account. Now two different users use the same agent: nothing in the request says whose the action is, so the server either acts with service-account privilege (sees *everyone's* calendar) or you try to patch it by passing `userId` as a tool argument. And `userId` as an argument is exactly the hole: it comes assembled by the LLM, it's manipulable by injection, and "swap the `userId`" becomes the exploit. An authenticated agent acting with the wrong user's privilege is privilege escalation — the worst kind, because Layer 1 logs everything as "a legitimate call from a trusted client".

**How the two layers flow together.** The correct path separates identity issuance from identity use:

1. **Agent authenticates (Layer 1).** The client opens the connection with its application Bearer token; the transport validates at the edge. From here the server knows *which agent* is speaking, but not yet *on whose behalf*.
2. **User authenticates and delegates (Layer 2).** Before acting for a person, the agent obtains a token representing *that* user — typically via OAuth with the user consenting to scopes ("this assistant may read your calendar"). That user token is what authorizes operations over their data.
3. **The call carries both identities.** The request arrives with the agent token (transport) and the user context (delegated). The server authorizes the *action* against the user token, not the agent's.
4. **The server derives the person from Layer 2, never from an argument.** Who the user is comes from the validated delegated token — never from a `userId`/`email` field the model filled in.

**Practical design rule.** Every tool that acts *on a specific person's data* needs both layers. Tools operating on global/public data (searching company docs, querying a catalog) can live with Layer 1 only. The review red flag: *does this tool's authorization depend on "who the user is"? If so, where does that identity come from?* If the answer is "from an argument", it's wrong.

**In multi-tenant, add a third dimension: the tenant.** The tenant must come from Layer 1 (the agent token identifies which organization that client belongs to) or from a trusted claim in the user token — never from an argument. That way you have: *which organization* (tenant, from the agent token), *which person* (user, from the delegated token), and *what they can do* (scopes). All three come from tokens validated at the edge; none comes from data that passed through the LLM.

### Observability — from day 1, not later

- **Log every call with the caller's identity:** tool name, input hash, latency, output size, error code, and *who* called. **Without logs, debugging LLM behavior is practically impossible** — and without identity in the log, you can neither audit access nor investigate token abuse.
- **Return errors as *structured error content*, not thrown exceptions** — so the LLM sees the error and decides whether to retry or escalate to the user. (Related auth caution: don't leak sensitive detail in the error message the model will read and possibly repeat to the user.)
- **Version it:** include a `version` field in the `initialize` response; breaking schema changes bump the major, and keep old schemas alive during the migration window.

> **Why auth is an architecture decision, not an implementation one:** the four points above determine where identity is resolved (edge vs. handler), how privilege is partitioned (scope), how user and agent relate (delegation), and what is trusted (nothing that passed through the LLM). Changing any of these later requires rewriting handlers and possibly the token model. It's cheap to get right on day 1 and expensive to fix on day 200.

## The 5 patterns

Each is a classic pattern + the correction for the LLM client. If you get the rules above right, you fall into them without needing the names. Summary first, detail after.

| Pattern | Ancestor | When to use | Main liability |
|---|---|---|---|
| **Resource Gateway** | Repository / REST | Expose read-mostly backend data, with sanitization | Complex joins/aggregations get awkward as resources |
| **Tool Orchestrator** | Facade / Mediator | Wrap a multi-system workflow in 1 composite tool | You own partial-failure handling; sub-tools lose reuse |
| **Stateful Session Server** | Session / Memento | Turn depends on prior state (open→edit→save) | State is invisible to the prompt; LLM may not pass the ID; leaks |
| **Proxy Aggregator** | Proxy / API gateway | Federate a fleet of servers under 1 endpoint | SPOF + extra hop; use the *scoped* variant, not *static-merge* |
| **Domain-Specific Adapter** | Adapter (GoF) | Wrap an "LLM-hostile" API (IDs, cryptic errors) | Overengineering if the API is already LLM-friendly |

**The two with a real platform decision:** *scoped Proxy Aggregator* (it's the mechanism that operationalizes rule 2's tool ceiling) and *Stateful Session Server* (because the "LLM-carries-the-ID" fragility is a risk to contain in design, not discover in production).

---

### 1. Resource Gateway
*Also called: Data Facade, Context Provider. Ancestor: Repository / REST resources.*

**Context.** An LLM agent needs to read structured data from one or more backends (databases, document stores, third-party APIs) and ground its answers in that data.

**Problem.** How do you expose backend data to an LLM in a way that's queryable, protected against prompt injection via untrusted data, and consistent when the backend schema changes?

**Solution.** Structure the server as a gateway that mediates all data access. Expose reads as **Resources** (list, get-by-id); turn it into a **Tool** when the query parameters would be unsafe in an open URI template. Insert a **sanitization layer** that strips/escapes injected content from backend responses *before* it reaches the LLM. In practice: the list handler exposes only metadata (ID, title, date) as URIs; the read handler resolves the URI, fetches the document, and runs the content through `sanitize()` before returning it.

**Consequences.** *Benefits:* single point of access-control enforcement; the LLM sees a stable interface even when the backend schema changes; prompt-injection risk is contained to one layer. *Liabilities:* an extra network hop on every read; backend schema changes propagate to the MCP server; complex joins/aggregations are awkward to express as resources.

**Known uses.** Database connectors (PostgreSQL, MongoDB), document-store bridges (Notion, Google Drive), REST API wrappers (GitHub, Jira, Linear).

**Reading.** The real value is being the *single point* of enforcement and the buffer that stabilizes the interface. It fits read-mostly key-value/document access; if your access is analytical with heavy joins, you'll fight the resource model — consider exposing parameterized queries as tools in that case.

---

### 2. Tool Orchestrator
*Also called: Action Hub, Workflow Facade. Ancestor: Facade / Mediator.*

**Context.** An LLM agent needs to perform actions that span multiple systems: e.g., create a ticket, notify the assignee, and post to a channel.

**Problem.** How do you expose multi-system workflows without requiring the LLM to understand each system's API, manage intermediate state across calls, or handle partial failure?

**Solution.** Expose **composite tools** that encapsulate the full workflow. Each tool makes all sub-calls internally and returns a single summary. The LLM sees one operation; the server does the orchestration. Example: a `create_and_notify_ticket` tool that internally creates the Jira issue, notifies the assignee in Slack, and fires the email — three API calls, one tool, one return summary.

**Consequences.** *Benefits:* reduces the LLM's reasoning load; enables transaction-like semantics for multi-step operations; hides API surface the LLM doesn't need to reason about. *Liabilities:* individual sub-tools become harder to reuse when the workflow changes; partial-failure handling becomes the server's responsibility, not the LLM's; the workflow logic now lives in two places (the tool + the docs).

**Known uses.** CI/CD automation servers, DevOps workflow tools, customer-support action hubs.

**Reading.** Central trade: you gain transactional semantics and reduce LLM load, but you own the rollback. If Slack fails after Jira creates, who undoes it? You do. Right pattern when the workflow is stable; it turns into debt when it changes every week. At scale, prefer few composite tools over many atomic ones — it helps respect the tool ceiling (rule 2) locally.

---

### 3. Stateful Session Server
*Also called: Conversational Context Server. Ancestor: Session / Memento.*

**Context.** An LLM agent conducts a multi-turn interaction where later calls depend on state established earlier: an open file, an in-flight database transaction, an authenticated user.

**Problem.** MCP tool calls are stateless request-response by default. How do you manage state that needs to persist across multiple calls within a session?

**Solution.** Generate a session ID on connection and include it in every tool response. Every subsequent call carries the session ID. The server keeps per-session context in memory (or Redis for horizontally-scaled deployments). Sessions expire on inactivity. Example: `open_file` reads the file and stores `{ filePath, content, edits: [] }` in the session map; `edit_file` retrieves that session by ID and validates that a file is open before applying the edit.

**Consequences.** *Benefits:* multi-turn workflows become natural; redundant data transfer is eliminated; transactional semantics are achievable. *Liabilities:* memory leak if sessions aren't reaped; horizontal scaling requires a distributed session store; the LLM has to pass the session ID reliably, which is **not guaranteed**.

**Known uses.** Code-editing agents (open → edit → save), database-transaction servers, multi-step form assistants.

**Reading.** The sharpest LLM delta in the paper: **the state is implicit, it's not in the prompt.** You're asking a probabilistic model to carry a state pointer. It's the pattern that fights scale the most — memory doesn't scale horizontally, it demands Redis + disciplined reaping, and the ID can get lost through an aggregator. Rule: use only when a turn genuinely depends on the previous one; minimize it; isolate it in a service with its own store. Note from §VI-A: statefulness is *invisible* in a function list — not even the taxonomy itself detects state by looking at the surface, only the code.

---

### 4. Proxy Aggregator
*Also called: MCP Router, Multi-Server Facade. Ancestor: Proxy / API gateway.*

**Context.** An LLM agent needs capabilities from many distinct MCP servers, but the client config limits how many connections it keeps, or an operator needs centralized auth and logging over a fleet.

**Problem.** How do you present multiple upstream servers as a single endpoint without losing per-server identity, versioning, or failure isolation?

**Solution.** A proxy connects to N upstream servers, **namespaces** tool names per server (to avoid collisions), and routes each call. Two variants — and the distinction is the point:

- **Static-merge:** exposes the union of all tools at once. Simplifies client config, but *increases the visible tool count* and degrades selection as soon as the merged catalog passes the ceiling (rule 2).
- **Scoped:** exposes only the subset relevant to the current task, retrieving candidates per request (**retrieval-over-tools**). Use whenever aggregation would push the context past the tool limit.

The mechanism: the proxy assembles the tool list from all upstreams, prefixing each name with the origin server's namespace (e.g., `github__create_pr`); when a call arrives, it splits off the prefix, finds the matching upstream, and routes the call with the original name.

**Consequences.** *Benefits:* simplifies client config; enables centralized auth and audit logging; supports tool discovery across a large fleet. *Liabilities:* introduces a single point of failure; adds a network hop to every call; namespace collisions require governance; upstream failures leak through the aggregate; and the scoped variant adds a per-request retrieval step that has to be fast *and* precise.

**Known uses.** Enterprise MCP gateways, dev-platform aggregators, multi-domain assistant backends.

**Reading.** It's the scale pattern — the one place you centralize auth, audit, and versioning for a fleet, and the only variant (scoped) that solves the tool ceiling. The trade that hurts: you swap "many tools in context" for "one retrieval on the hot path". If retrieval misses, the right tool never even enters the context — an invisible failure, worse than a mis-selected tool. At scale, **the quality of tool retrieval is what determines whether the platform scales or collapses.**

---

### 5. Domain-Specific Adapter
*Also called: Semantic Layer, Domain Translator. Ancestor: Adapter (GoF).*

**Context.** An existing system has a useful but "LLM-hostile" API: machine-readable identifiers, low-level operations, complex auth flows, or output formats that need substantial post-processing.

**Problem.** How do you translate a complex, low-level API into a form the LLM uses accurately, without reimplementing the business logic in the server?

**Solution.** Build a semantic adapter that wraps the existing API and adds four things: **human-readable descriptions** that guide LLM selection; **input normalization** (accept natural-language dates/names, fuzzy IDs); **output enrichment** (resolve IDs to display names); and **error translation** (API code → clear plain language).

**Consequences.** *Benefits:* LLM selection accuracy improves when the descriptions are precise; API complexity is isolated in the adapter; backend versioning can be absorbed in the adapter layer. *Liabilities:* the adapter must be updated when the underlying API changes; overengineering is a real risk when the API is already LLM-friendly.

**Known uses.** CRM adapters (Salesforce, HubSpot), financial-data connectors, healthcare record systems.

**Reading.** The one that gains the most value at scale: it's the *seam* that absorbs upstream API churn — the LLM-facing surface stays stable even when the backend changes. At scale, where you don't control every API, it's what protects the model from churn propagating through. The real liability: not every API needs an adapter — if the API already has semantic names and readable errors, the adapter is dead weight you maintain for nothing.

## Review checklist

- [ ] Does every tool have a description that explains what / when / what it returns?
- [ ] Does no context expose more than ~10–15 tools?
- [ ] Does a large catalog use retrieval-over-tools, not static-merge?
- [ ] No "God Tool" with a free `action: string` / `params: object`?
- [ ] Does all external content pass through named sanitization before reaching the LLM?
- [ ] Do long operations return `job_id` + `poll_job` instead of blocking?
- [ ] State only where needed, with expiration/reaping planned?
- [ ] Server co-located when latency matters; network hops counted?
- [ ] Auth at the transport; every call logged with identity?
- [ ] `version` in `initialize`; old schemas alive during migration?

## What this paper does NOT teach you

It **assumes** you already know MCP. It's not a source for learning the protocol (primitives, the `initialize` handshake, capability negotiation) — for that, the official spec at modelcontextprotocol.io. The value here is architectural: how to **structure** a server, not how the protocol works.
