# Aprendizados — *MCP Server Architecture Patterns for LLM-Integrated Applications*

> **Rodrigues & Vas** · arXiv:[2606.30317](https://arxiv.org/abs/2606.30317) · candidato ICSME 2026
> **Ângulo:** construir um bom servidor MCP — não estudar o paper.
> **Recorte:** filtrado pelo que tem consequência de engenharia; κ, inter-rater e formalismo de padrões ficaram de fora.

---

## TL;DR

Projetar um servidor MCP é um problema de *API design* com uma restrição incomum: **o cliente é um LLM que escolhe a operação lendo a descrição em linguagem natural em runtime, não consultando o schema nem a doc.** Quase toda decisão de design boa deriva dessa única restrição. O protocolo é deliberadamente minimalista — estado, async e sanitização ficam por sua conta.

---

## As 5 regras que mais importam pra construir

Em ordem de impacto.

### 1. Descrição de tool é artefato *load-bearing* — trate como código

É o tópico central do paper. O LLM seleciona a tool pela descrição, não pelo schema. Descrição vaga (ou que só repete o nome, tipo `send_message` sem texto) faz a tool **funcionar no teste e não ser selecionada em produção** — o modo de falha mais insidioso, porque passa em teste unitário.

- Escreva: o que faz, quando usar, o que retorna — para alguém que nunca viu.
- Revise a descrição em PR, com o mesmo rigor do código.
- Trate divergência entre descrição e comportamento como **bug**, não como doc desatualizada.

### 2. Mantenha ≤ ~10–15 tools por contexto

Teto prático de acurácia de seleção medido em produção:

| Modelo | Cai abaixo de 90% de acurácia |
|---|---|
| Claude Haiku 4.5 | entre **10 e 15** tools (91% @ 10, 87% @ 15) |
| Claude Sonnet 4 | entre **20 e 30** tools (≥90% até 20) |

Consequência de design: **não jogue todas as tools num servidor só.** Se o catálogo cresce, particione e faça **retrieval-over-tools** — exponha só o subconjunto relevante por request — em vez de *static-merge* de tudo. Menos tools bem-nomeadas > muitas tools ambíguas.

> ⚠️ Ressalva: o número vem de telemetria observacional de **uma** empresa de voz-AI com budget de latência apertado (500ms). Trate 10–15 como **piso conservador para caminhos latency-sensitive**, não lei universal. O mecanismo (partição + retrieval) generaliza; o número específico não. Trabalhos independentes (RAG-MCP; LongFuncEval) confirmam a degradação, com o joelho da curva por volta de ~30 candidatas em cenários sem restrição de latência.

### 3. Decomponha — nunca crie uma "God Tool"

Cada operação distinta = uma tool nomeada com schema preciso. Nunca `do_anything(action: string, params: object)`: esconder a semântica dentro de um argumento de runtime destrói o sinal de seleção (é a regra 1 ao contrário). Nem tools demais (regra 2), nem uma tool que deveria ser dez.

#### Ruim — a God Tool clássica

```json
{
  "name": "manage_calendar",
  "description": "Gerencia eventos do calendário",
  "inputSchema": {
    "type": "object",
    "properties": {
      "action": { "type": "string" },
      "params": { "type": "object" }
    }
  }
}
```

O LLM tem que *adivinhar* que `action` aceita `"create"|"update"|"delete"|"list"` e que `params` muda de forma conforme a action. Nada disso está no sinal que ele usa pra decidir (nome + descrição). Ele chama `manage_calendar` pra qualquer coisa de calendário e depois erra o `params`. Falha dupla: seleção certa, argumentos errados.

#### Bom — decomposto por operação

```json
{
  "name": "create_calendar_event",
  "description": "Cria um novo evento no calendário do usuário. Use quando o usuário quer agendar algo novo (reunião, lembrete, compromisso). Retorna o ID e o link do evento criado. Não use para modificar eventos existentes — use update_calendar_event.",
  "inputSchema": {
    "type": "object",
    "required": ["title", "start", "end"],
    "properties": {
      "title":  { "type": "string", "description": "Título do evento" },
      "start":  { "type": "string", "format": "date-time" },
      "end":    { "type": "string", "format": "date-time" },
      "attendees": { "type": "array", "items": { "type": "string", "format": "email" } }
    }
  }
}
```

Mais `update_calendar_event`, `delete_calendar_event`, `list_calendar_events` no mesmo estilo. Cada tool tem schema tipado (o modelo sabe que `start` é date-time, não um blob livre) e a descrição diz *quando usar* e *quando não usar*, apontando pra tool irmã. É a regra 1 e a regra 3 trabalhando juntas.

#### A God Tool sutil — a que engana

Passa em review porque *parece* decomposto:

```json
{
  "name": "search",
  "description": "Busca no sistema",
  "inputSchema": {
    "type": "object",
    "properties": {
      "type":  { "type": "string", "enum": ["user", "order", "product", "invoice"] },
      "query": { "type": "string" }
    }
  }
}
```

Tem enum, tem schema tipado — mas ainda é uma God Tool disfarçada. O campo `type` **ramifica o comportamento em runtime**, então o LLM raciocina sobre semântica de negócio (buscar user vs. invoice são intenções diferentes) dentro de um argumento, não na escolha da tool. O sinal de seleção continua fraco.

Versão boa:

```json
{ "name": "search_users",    "description": "Busca usuários por nome, email ou ID. Retorna perfil e status da conta." }
{ "name": "search_orders",   "description": "Busca pedidos por número, cliente ou data. Retorna itens, valor e status de entrega." }
{ "name": "search_invoices", "description": "Busca faturas por número ou cliente. Retorna valor, vencimento e status de pagamento." }
```

#### A regra que decide — comportamento vs. filtro

Não decomponha ao ponto de estourar o teto de tools (regra 2). A linha divisória:

> **O argumento muda *o que a tool faz* (→ decompõe) ou só *filtra o mesmo resultado* (→ vira parâmetro)?**

- **Decompõe** quando as operações têm intenções, schemas de input ou retornos diferentes → `create_event` vs. `delete_event`. No `manage_calendar`, `action` mudava o que a tool faz — errado.
- **Não decompõe** quando é o mesmo input variando só num valor sem mudar semântica. Não faça `list_events_today` / `list_events_this_week` / `list_events_this_month` — isso é uma tool `list_events(range)`, porque intenção e retorno são idênticos e o `range` só filtra — certo.

O teste rápido para o caso sutil: *se os "tipos"/"actions" têm descrições, casos de uso ou campos de retorno diferentes, são tools diferentes.*

### 4. Sanitize todo conteúdo externo antes de retornar

Único item de **segurança** da lista. Conteúdo de backend/usuário (comentário, corpo de documento, input de form) chega cru ao LLM, e um texto com *"Ignore previous instructions and…"* é processado como **instrução, não dado** (prompt injection). Insira uma camada de sanitização **explícita e nomeada** — não confie no backend, não faça inline e esqueça. O protocolo não faz isso por você.

### 5. Nunca exponha operação síncrona longa

MCP não tem mecanismo de callback assíncrono. Qualquer operação acima de poucos segundos (encoding de vídeo, processamento de arquivo grande) faz o cliente dar **timeout**. Padrão canônico: retorne um `job_id` de forma síncrona e exponha uma tool separada `poll_job(id)`.

## 3 decisões de estrutura

### Estado — só quando necessário, com budget pra *reaping*

MCP é **stateless request-response por padrão**. Se você precisa de estado entre chamadas:

- Gere um `session_id` na conexão, inclua em toda resposta, e o cliente carrega de volta em cada chamada.
- Guarde contexto por-sessão em memória — ou Redis pra escalar horizontal.
- Expire sessões por inatividade.

Custos reais que você herda: (a) o **LLM precisa passar o session ID de forma confiável, o que não é garantido** — você está pedindo a um modelo probabilístico pra carregar um ponteiro de estado; (b) memória vaza se as sessões não forem reapadas; (c) escala horizontal exige store distribuído. **Adicione estado só quando um turn genuinamente depende do anterior**, não por conveniência.

### Co-localização importa mais que a escolha de transporte

Dois transportes: `stdio` (local, in-process) e `streamable-http` (remoto, HTTP + SSE opcional). A diferença de overhead entre eles é **sub-milissegundo e irrelevante assim que você cruza fronteira de host** — a RTT de rede (~30ms same-region) domina em 2–3 ordens de grandeza.

As decisões que de fato mexem na latência:

1. O servidor está **co-localizado** com o cliente?
2. O fan-out downstream (um proxy/aggregator) adiciona **outro hop de rede**? Cada camada de proxy dobra a exposição à RTT.

Decida isso primeiro; o encoding do transporte é detalhe.

### Autenticação — decida antes de escrever a primeira tool

Auth em servidor MCP não é "adicionar um middleware depois". A restrição do cliente-LLM muda o modelo de ameaça: o cliente não é um humano nem um app determinístico, é um modelo que decide o que chamar em runtime a partir de texto que pode ter sido injetado. Por isso auth precisa estar na base do design, não no fim.

**Regra 1 — autentique no transporte, nunca dentro do handler de tool.** `streamable-http` suporta **Bearer token** no header HTTP. A verificação acontece *antes* de o request virar uma tool call. Se você deixa a checagem de auth dentro do handler (`if (!isAuthorized(req)) throw...`), você está confiando que toda tool implementou a checagem certo — e basta uma esquecer. Pior: um handler que autentica olhando argumentos do request está a um prompt injection de ser enganado, porque os argumentos vêm do LLM. Auth na borda, identidade resolvida antes de qualquer lógica de negócio.

**Regra 2 — escope tokens por conjunto de tools, não por servidor inteiro.** Um token não deve dar acesso a tudo que o servidor expõe. Modele escopos como capabilities: um token de leitura acessa só as tools/resources de leitura; um token de escrita, um subconjunto explícito. Isso limita o *blast radius* quando um token vaza (e tokens vazam — ficam em config de cliente, em logs, em prints). Em multi-tenant, o escopo também carrega o tenant: o token identifica *qual* tenant, e o servidor nunca deriva tenant de um argumento passado pelo LLM.

**Regra 3 — a identidade autenticada é do chamador, não do usuário final.** Aqui mora a confusão mais comum. O que se autentica no transporte é a *aplicação cliente / agente*, não necessariamente a pessoa por trás dela. Se suas tools agem em nome de um usuário (ler o calendário *dele*, mandar email *como* ele), você precisa de uma segunda camada: o token do transporte diz "este agente pode falar comigo"; um contexto de usuário (OAuth delegado, token de sessão do usuário repassado) diz "e ele está agindo por conta da pessoa X". Não colapse os dois. Um agente autenticado agindo com privilégio de usuário errado é escalonamento de privilégio.

**Regra 4 — auth nunca depende de conteúdo que passou pelo LLM.** Decisão de autorização se baseia em: o token (transporte), o escopo (o que ele permite) e, quando aplicável, o contexto de usuário delegado. *Nunca* num argumento da tool, num campo do request montado pelo modelo, ou em texto vindo de um resource. Tudo que atravessa o LLM é potencialmente adversarial. Este é o ponto onde auth em MCP difere de auth em API tradicional — o "cliente" é manipulável por injeção.

**stdio é um modelo de confiança diferente.** No transporte `stdio` o servidor roda como subprocesso local do cliente, herdando as permissões do processo/usuário que o lançou — não há Bearer token no meio. A fronteira de segurança ali é o SO, não o protocolo. Isso é adequado pra ferramentas locais (acesso a arquivos do próprio usuário), mas significa que "mover pra remoto" (streamable-http) não é só trocar transporte: é adicionar toda a camada de auth que o stdio te deixava pular. Decida cedo se o servidor será remoto, porque isso muda o design de segurança inteiro.

**Onde o Proxy Aggregator entra.** Em escala, o aggregator é o lugar natural pra centralizar auth: um ponto único de emissão/validação de token, de mapeamento de escopo e de audit do fleet inteiro, em vez de cada servidor upstream reinventar. O custo é que ele vira também um alvo concentrado — comprometer o aggregator compromete o acesso ao fleet. Trate-o como componente de segurança de primeira classe (rotação de credenciais, isolamento, HA), não como um roteador burro.

### As duas camadas de identidade — agente vs. usuário

Este é o erro de auth mais caro de corrigir depois, porque colapsar as duas camadas parece funcionar em desenvolvimento e só explode quando há mais de um usuário real. Separe-as desde o início.

**Camada 1 — o agente pode falar comigo?** É o Bearer token do transporte (seção acima). Ele autentica a *aplicação cliente / agente* — o processo que abriu a conexão MCP. Responde: "esta coisa que está me chamando é um cliente que eu reconheço e autorizo a conversar?". É binário e por-conexão.

**Camada 2 — ele está agindo por conta de quem?** É o **contexto de usuário delegado**: um OAuth em nome do usuário, ou um token de sessão do usuário repassado na chamada. Responde: "e as ações que ele está pedindo são em nome da pessoa X, com os privilégios *da pessoa X*?". É por-usuário e muda a cada pessoa que o agente serve.

> O token do transporte diz **"este agente pode falar comigo"**. O contexto de usuário diz **"e ele está agindo por conta da pessoa X"**. Não colapse os dois.

**Por que colapsar é escalonamento de privilégio.** Imagine um assistente que lê o calendário e manda email. Se você só tem a Camada 1, o agente autentica com *um* token e todas as tools rodam com *um* privilégio — provavelmente o de um service account amplo. Agora dois usuários diferentes usam o mesmo agente: nada no request diz de quem é a ação, então o servidor ou age com privilégio de service account (vê o calendário de *todo mundo*) ou você tenta remendar passando o `userId` como argumento da tool. E `userId` como argumento é justamente o buraco: vem montado pelo LLM, é manipulável por injeção, e "trocar o `userId`" vira o exploit. Um agente autenticado agindo com o privilégio do usuário errado é escalonamento de privilégio — o pior tipo, porque a Camada 1 registra tudo como "chamada legítima de um cliente confiável".

**Como as duas camadas fluem juntas.** O caminho correto separa emissão de identidade de uso de identidade:

1. **Agente autentica (Camada 1).** O cliente abre a conexão com seu Bearer token de aplicação; o transporte valida na borda. A partir daqui o servidor sabe *qual agente* está falando, mas ainda não *por quem*.
2. **Usuário se autentica e delega (Camada 2).** Antes de agir por uma pessoa, o agente obtém um token que representa *aquele* usuário — tipicamente via OAuth com o usuário consentindo os escopos ("este assistente pode ler seu calendário"). Esse token do usuário é o que autoriza operações sobre os dados dele.
3. **A chamada carrega as duas identidades.** O request chega com o token de agente (transporte) e o contexto de usuário (delegado). O servidor autoriza a *ação* contra o token do usuário, não contra o do agente.
4. **O servidor deriva a pessoa da Camada 2, jamais de um argumento.** Quem é o usuário sai do token delegado validado — nunca de um campo `userId`/`email` que o modelo preencheu.

**Regra prática de design.** Toda tool que age *sobre dados de uma pessoa específica* precisa das duas camadas. Tools que operam sobre dados globais/públicos (buscar na documentação da empresa, consultar um catálogo) podem viver só com a Camada 1. O sinal de alerta em review: *a autorização desta tool depende de "quem é o usuário"? Se sim, de onde vem essa identidade?* Se a resposta for "de um argumento", está errado.

**Em multi-tenant, some uma terceira dimensão: o tenant.** O tenant deve sair da Camada 1 (o token do agente identifica de qual organização é aquele cliente) ou de um claim confiável no token do usuário — nunca de argumento. Assim você tem: *qual organização* (tenant, do token de agente), *qual pessoa* (usuário, do token delegado) e *o que ela pode* (escopos). As três vêm de tokens validados na borda; nenhuma vem de dado que passou pelo LLM.

### Observabilidade — desde o dia 1, não depois

- **Logue toda chamada com identidade do chamador:** nome da tool, hash do input, latência, tamanho do output, error code, e *quem* chamou. **Sem logs, debugar comportamento de LLM é praticamente impossível** — e sem identidade no log, você não consegue nem auditar acesso nem investigar abuso de token.
- **Retorne erros como *structured error content*, não exceções lançadas** — assim o LLM vê o erro, decide se re-tenta ou escala pro usuário. (Cuidado correlato de auth: não vaze detalhe sensível na mensagem de erro que o modelo vai ler e possivelmente repetir ao usuário.)
- **Versione:** inclua um campo `version` no response de `initialize`; breaking changes em schema incrementam a major, e mantenha schemas antigos vivos durante a janela de migração.

> **Por que auth é decisão de arquitetura, não de implementação:** os quatro pontos acima determinam onde a identidade é resolvida (borda vs. handler), como o privilégio é particionado (escopo), como usuário e agente se relacionam (delegação) e o que é confiável (nada que passou pelo LLM). Trocar qualquer um desses depois exige reescrever handlers e possivelmente o modelo de tokens. É barato acertar no dia 1 e caro corrigir no dia 200.

## Os 5 padrões

Cada um é um padrão clássico + a correção pro cliente-LLM. Se você acertar as regras acima, cai neles sem precisar dos nomes. Resumo primeiro, detalhe em seguida.

| Padrão | Ancestral | Quando usar | Liability principal |
|---|---|---|---|
| **Resource Gateway** | Repository / REST | Expor dados read-mostly de backend, com sanitização | Joins/agregações complexas ficam esquisitas como resources |
| **Tool Orchestrator** | Facade / Mediator | Empacotar workflow multi-sistema em 1 tool composta | Você assume o tratamento de falha parcial; sub-tools perdem reuso |
| **Stateful Session Server** | Session / Memento | Turn depende de estado anterior (open→edit→save) | Estado é invisível ao prompt; LLM pode não passar o ID; leaks |
| **Proxy Aggregator** | Proxy / API gateway | Federar um fleet de servidores sob 1 endpoint | SPOF + hop extra; use a variante *scoped*, não *static-merge* |
| **Domain-Specific Adapter** | Adapter (GoF) | Envolver API "hostil a LLM" (IDs, erros crípticos) | Overengineering se a API já é LLM-friendly |

**Os dois com decisão de plataforma real:** *Proxy Aggregator scoped* (é o mecanismo que operacionaliza o limite de tools da regra 2) e *Stateful Session Server* (porque a fragilidade "LLM-carrega-o-ID" é risco a conter em design, não a descobrir em produção).

### 1. Resource Gateway
*Também chamado: Data Facade, Context Provider. Ancestral: Repository / REST resources.*

**Contexto.** Um agente LLM precisa ler dados estruturados de um ou mais backends (bancos, document stores, APIs de terceiros) e fundamentar as respostas nesses dados.

**Problema.** Como expor dados de backend a um LLM de forma que seja consultável, protegida contra prompt injection via dados não-confiáveis, e consistente quando o schema do backend muda?

**Solução.** Estruture o servidor como um gateway que media todo acesso a dados. Exponha leituras como **Resources** (list, get-by-id); vire **Tool** quando os parâmetros de query seriam inseguros num template de URI aberto. Insira uma **camada de sanitização** que remove/escapa conteúdo injetado das respostas do backend *antes* de chegar ao LLM. Na prática: o handler de listagem expõe só metadados (ID, título, data) como URIs; o handler de leitura resolve o URI, busca o documento e passa o conteúdo por `sanitize()` antes de devolver.

**Consequências.** *Benefícios:* ponto único de enforcement de access control; o LLM vê uma interface estável mesmo quando o schema do backend muda; o risco de prompt injection fica contido numa camada. *Liabilities:* um hop de rede extra em toda leitura; mudanças de schema no backend se propagam pro servidor MCP; joins/agregações complexas ficam esquisitas de expressar como resources.

**Usos conhecidos.** Conectores de banco (PostgreSQL, MongoDB), pontes de document store (Notion, Google Drive), wrappers de API REST (GitHub, Jira, Linear).

**Leitura.** O valor real é ser o *único ponto* de enforcement e o buffer que estabiliza a interface. Encaixa em acesso read-mostly chave-valor/documento; se seu acesso é analítico com joins pesados, você vai brigar com o modelo de resources — considere expor queries parametrizadas como tools nesse caso.

---

### 2. Tool Orchestrator
*Também chamado: Action Hub, Workflow Facade. Ancestral: Facade / Mediator.*

**Contexto.** Um agente LLM precisa executar ações que atravessam múltiplos sistemas: por exemplo, criar um ticket, notificar o responsável e postar num canal.

**Problema.** Como expor workflows multi-sistema sem exigir que o LLM entenda a API de cada sistema, gerencie estado intermediário entre chamadas, ou trate falha parcial?

**Solução.** Exponha **tools compostas** que encapsulam o workflow completo. Cada tool faz todas as sub-chamadas internamente e retorna um único resumo. O LLM vê uma operação; o servidor faz a orquestração. Exemplo: uma tool `create_and_notify_ticket` que internamente cria a issue no Jira, notifica o responsável no Slack e dispara o email — três chamadas de API, uma única tool, um resumo de retorno.

**Consequências.** *Benefícios:* reduz a carga de raciocínio do LLM; permite semântica transação-like para operações multi-step; esconde superfície de API que o LLM não precisa raciocinar. *Liabilities:* sub-tools individuais ficam mais difíceis de reusar quando o workflow muda; tratamento de falha parcial vira responsabilidade do servidor, não do LLM; a lógica do workflow passa a viver em dois lugares (a tool + a doc).

**Usos conhecidos.** Servidores de automação CI/CD, tools de workflow DevOps, action hubs de suporte ao cliente.

**Leitura.** Trade central: você ganha semântica transacional e reduz carga do LLM, mas assume o rollback. Se o Slack falha depois do Jira criar, quem desfaz? É seu. Padrão certo quando o workflow é estável; vira dívida quando ele muda toda semana. Em escala, prefira poucas tools compostas a muitas atômicas — ajuda a respeitar o teto de tools (regra 2) localmente.

### 3. Stateful Session Server
*Também chamado: Conversational Context Server. Ancestral: Session / Memento.*

**Contexto.** Um agente LLM conduz uma interação multi-turn onde chamadas posteriores dependem de estado estabelecido antes: um arquivo aberto, uma transação de banco em andamento, um usuário autenticado.

**Problema.** Tool calls do MCP são request-response stateless por padrão. Como gerenciar estado que precisa persistir entre múltiplas chamadas dentro de uma sessão?

**Solução.** Gere um session ID na conexão e inclua em toda resposta de tool. Toda chamada seguinte carrega o session ID. O servidor mantém contexto por-sessão em memória (ou Redis para deployments com escala horizontal). Sessões expiram por inatividade. Exemplo: `open_file` lê o arquivo e guarda `{ filePath, content, edits: [] }` no mapa de sessões; `edit_file` recupera essa sessão pelo ID e valida que há um arquivo aberto antes de aplicar a edição.

**Consequências.** *Benefícios:* workflows multi-turn ficam naturais; transferência redundante de dados é eliminada; semântica transacional é alcançável. *Liabilities:* memory leak se as sessões não forem reapadas; escala horizontal exige store de sessão distribuído; o LLM precisa passar o session ID de forma confiável, o que **não é garantido**.

**Usos conhecidos.** Agentes de edição de código (open → edit → save), servidores de transação de banco, assistentes de formulário multi-step.

**Leitura.** O delta LLM mais afiado do paper: **o estado é implícito, não está no prompt.** Você está pedindo a um modelo probabilístico pra carregar um ponteiro de estado. É o padrão que mais briga com escala — memória não escala horizontal, exige Redis + reaping disciplinado, e o ID pode se perder através de um aggregator. Regra: use só quando um turn genuinamente depende do anterior; minimize; isole num serviço com store próprio. Nota do §VI-A: statefulness é *invisível* numa lista de funções — nem a própria taxonomia detecta estado olhando a superfície, só o código.

### 4. Proxy Aggregator
*Também chamado: MCP Router, Multi-Server Facade. Ancestral: Proxy / API gateway.*

**Contexto.** Um agente LLM precisa de capacidades de muitos servidores MCP distintos, mas a config do cliente limita quantas conexões ele mantém, ou um operador precisa de auth e logging centralizados sobre um fleet.

**Problema.** Como apresentar múltiplos servidores upstream como um único endpoint sem perder identidade por-servidor, versionamento ou isolamento de falha?

**Solução.** Um proxy conecta a N servidores upstream, faz **namespacing** dos nomes de tool por servidor (pra evitar colisão) e roteia cada chamada. Duas variantes — e a distinção é o ponto:

- **Static-merge:** expõe a união de todas as tools de uma vez. Simplifica a config do cliente, mas *aumenta a contagem visível de tools* e degrada a seleção assim que o catálogo mergeado passa o teto (regra 2).
- **Scoped:** expõe só o subconjunto relevante à tarefa atual, recuperando candidatos por request (**retrieval-over-tools**). Use sempre que a agregação empurraria o contexto além do limite de tools.

O mecanismo: o proxy monta a lista de tools de todos os upstreams prefixando cada nome com o namespace do servidor de origem (ex.: `github__create_pr`); quando uma chamada chega, ele separa o prefixo, encontra o upstream correspondente e roteia a chamada com o nome original.

**Consequências.** *Benefícios:* simplifica a config do cliente; permite auth e audit logging centralizados; suporta descoberta de tools num fleet grande. *Liabilities:* introduz um single point of failure; adiciona um hop de rede a cada chamada; colisões de namespace exigem governança; falhas de upstream vazam pelo agregado; e a variante scoped adiciona um passo de retrieval por request que precisa ser rápido *e* preciso.

**Usos conhecidos.** Gateways MCP corporativos, agregadores de plataforma de dev, backends de assistente multi-domínio.

**Leitura.** É o padrão de escala — o único lugar onde você centraliza auth, audit e versionamento de um fleet, e a única variante (scoped) que resolve o teto de tools. O trade que dói: você troca "muitas tools no contexto" por "um retrieval no hot path". Se o retrieval erra, a tool certa nem entra no contexto — falha invisível, pior que tool mal-selecionada. Em escala, **a qualidade do retrieval de tools é o que determina se a plataforma escala ou desmorona.**

### 5. Domain-Specific Adapter
*Também chamado: Semantic Layer, Domain Translator. Ancestral: Adapter (GoF).*

**Contexto.** Um sistema existente tem uma API útil mas "hostil a LLM": identificadores machine-readable, operações low-level, fluxos de auth complexos, ou formatos de output que exigem pós-processamento substancial.

**Problema.** Como traduzir uma API complexa e low-level numa forma que o LLM use com precisão, sem reimplementar a lógica de negócio no servidor?

**Solução.** Construa um adapter semântico que envolve a API existente e adiciona quatro coisas: **descrições human-readable** que guiam a seleção do LLM; **normalização de input** (aceitar datas/nomes em linguagem natural, IDs fuzzy); **enriquecimento de output** (resolver IDs pra nomes de exibição); e **tradução de erro** (código de API → inglês/português claro).

**Consequências.** *Benefícios:* a acurácia de seleção do LLM melhora quando as descrições são precisas; a complexidade da API fica isolada no adapter; versionamento do backend pode ser absorvido na camada do adapter. *Liabilities:* o adapter precisa ser atualizado quando a API subjacente muda; overengineering é risco real quando a API já é LLM-friendly.

**Usos conhecidos.** Adapters de CRM (Salesforce, HubSpot), conectores de dados financeiros, sistemas de prontuário de saúde.

**Leitura.** O que mais ganha valor em escala: é o *seam* que absorve churn de API upstream — a superfície LLM-facing fica estável mesmo quando o backend muda. Em escala, onde você não controla todas as APIs, é o que protege o modelo de churn se propagando. A liability real: nem toda API precisa de adapter — se a API já tem nomes semânticos e erros legíveis, o adapter é camada morta que você mantém à toa.

## Checklist de review

- [ ] Toda tool tem descrição que explica o quê / quando / o que retorna?
- [ ] Nenhum contexto expõe mais de ~10–15 tools?
- [ ] Catálogo grande usa retrieval-over-tools, não static-merge?
- [ ] Nenhuma "God Tool" com `action: string` / `params: object` livre?
- [ ] Todo conteúdo externo passa por sanitização nomeada antes de ir ao LLM?
- [ ] Operações longas retornam `job_id` + `poll_job`, não bloqueiam?
- [ ] Estado só onde é necessário, com expiração/reaping planejados?
- [ ] Servidor co-localizado quando latência importa; hops de rede contados?
- [ ] Auth no transporte; toda chamada logada com identidade?
- [ ] `version` no `initialize`; schemas antigos vivos na migração?
